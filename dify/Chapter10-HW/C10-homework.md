# Chapter 10 作业：基于 Dify 构建代码生成助手 Pro

> **作业目标**：在 Dify 平台上完成「代码生成助手 Pro」应用的开发，并通过该应用成功生成贪吃蛇游戏代码与 DeepSeek 官网仿制代码。

---

## 一、应用概述

「代码生成助手 Pro」是一个基于 Dify 工作流构建的多节点 AI 应用，其核心能力是将**模糊的自然语言需求**逐步转化为**可运行的代码**。

整个应用由三个串联的 LLM 节点组成，分别承担不同的职责：

```
用户输入需求
    │
    ▼
┌─────────────────┐
│  节点 1：需求设计  │  ← 产品架构师角色，补全 User Story、模块拆解、I/O 流程
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  节点 2：需求整理  │  ← 资深 BA 角色，提取功能清单、业务规则、数据结构
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  节点 3：需求开发  │  ← 资深工程师角色，输出完整可运行代码
└─────────────────┘
```

---

## 二、关键步骤记录

### 2.1 创建 Dify 工作流应用

在 Dify 控制台新建一个 **Workflow（工作流）** 类型的应用，命名为「代码生成助手 Pro」。

![代码生成助手 Pro 工作流全貌](images/代码生成助手Pro.png)

工作流由以下节点构成：

| 节点 | 角色定位 | 输入变量 | 输出变量 |
|------|---------|---------|---------|
| 开始节点 | — | `sys.query`（用户需求描述） | — |
| LLM 节点 1：需求设计 | 产品架构师 | `sys.query` | `design_output` |
| LLM 节点 2：需求整理 | 资深 BA | `design_output` | `spec_output` |
| LLM 节点 3：需求开发 | 资深工程师 | `spec_output` | `code_output` |
| 结束节点 | — | `code_output` | 最终代码 |

---

### 2.2 配置各节点 System Prompt

每个 LLM 节点均配置了专属的 System Prompt，以确保 AI 在对应阶段扮演正确的角色。

#### 节点 1：需求设计（产品架构师）

> 完整 Prompt 见：[`docs/LLM-需求设计-System-Prompt.md`](docs/LLM-需求设计-System-Prompt.md)

**核心职责**：
- 基于用户的简短描述，补全使用路径（User Story）
- 拆解功能模块
- 设计输入-处理-输出（I/O）流程
- 评估潜在技术风险与性能瓶颈

**关键变量引用**：节点通过 `{{#sys.query#}}` 接收用户原始输入。

---

#### 节点 2：需求整理（资深 BA）

> 完整 Prompt 见：[`docs/LLM-需求整理-System-Prompt.md`](docs/LLM-需求整理-System-Prompt.md)

**核心职责**：
- 从上一节点的设计文档中提取核心目标
- 输出标准化功能清单
- 整理业务逻辑规则与数据结构建议
- 标注待确认的模糊事项

**关键变量引用**：通过 `{{#sys.files#}}` 接收上游节点的设计输出。

---

#### 节点 3：需求开发（资深工程师）

> 完整 Prompt 见：[`docs/LLM-需求开发-System-Prompt.md`](docs/LLM-需求开发-System-Prompt.md)

**核心职责**：
- 依据整理好的需求规格，输出完整、可运行的代码
- 代码需包含必要注释，结构清晰

---

### 2.3 节点间变量传递

Dify 工作流中，节点间通过**变量引用**实现数据流转。配置方式如下：

1. 在节点 2 的 User Prompt 中，通过 `/` 或 `{` 触发变量选择器，引用节点 1 的输出变量
2. 在节点 3 的 User Prompt 中，同样引用节点 2 的输出变量
3. 结束节点配置输出变量，指向节点 3 的代码输出

## 五、总结与反思

### 5.1 工作流设计的价值

通过将代码生成拆分为「需求设计 → 需求整理 → 需求开发」三个阶段，每个 LLM 节点专注于单一职责，有效避免了直接让 AI 生成代码时常见的**需求理解偏差**问题。

```
传统方式：用户需求 ──────────────────────────────→ 代码（质量不稳定）

工作流方式：用户需求 → 设计文档 → 规格说明 → 代码（质量可控）
```

### 5.2 关键经验

1. **System Prompt 的角色设定至关重要**：为每个节点赋予明确的专业角色（架构师、BA、工程师），能显著提升输出质量
2. **变量传递是工作流的核心**：上下游节点通过变量引用形成完整的信息流，每个节点都在前一节点的基础上深化处理
3. **PRD 文档作为上下文输入**：对于复杂项目（如官网），提前准备结构化的需求文档，能让代码生成节点产出更贴近预期的结果

### 5.3 可优化方向

- 在节点 3（需求开发）中增加**代码审查**子节点，对生成代码进行自动化质量检查
- 引入**条件分支**节点，根据需求复杂度动态决定是否启用深度设计阶段
- 结合 Dify 的**工具节点**，实现代码的自动化运行与测试验证
