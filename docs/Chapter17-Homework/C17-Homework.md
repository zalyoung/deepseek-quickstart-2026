# 权限系统设计与数据隐私保护作业

## 1. 内部 Agent 应用的权限系统设计

为了确保企业内部 Agent 应用的安全，我会采用 **RBAC（基于角色的访问控制）** 模型。将复杂的权限细分，根据员工的职能来定义不同的角色。

### 定义的角色与权限

* **管理员（Admin）：** * **权限：** 拥有“全家桶”最高权限。包括用户管理、系统参数配置、审计日志查看，以及所有模型和知识库的增删改查。
* **原因：** 系统需要一个总负责人来把控整体运行和安全。


* **开发者（Developer）：**
* **权限：** 侧重于“干活”的权限。可以调用 API、编排 Agent 工作流、接入 MCP 协议工具，以及进行模型的微调与测试。
* **原因：** 开发者需要自由度来调试 Agent，但通常不需要接触过多的敏感业务数据。


* **普通用户（User）：**
* **权限：** 仅拥有“使用”权限。可以在前端与 Agent 对话，查询授权范围内的知识库（RAG），但不能修改模型配置或查看他人日志。
* **原因：** 保证业务部门能够正常使用 AI 工具，同时防止误操作影响系统稳定。



### 核心权限维度

我会将权限拆解为三个维度进行交叉控制：

1. **模型访问：** 谁能用 GPT-4，谁只能用轻量化的小模型。
2. **数据读写：** 区分对企业知识库的只读（查询）和写入权限。
3. **API 调用：** 严格限制外部插件或工具的触发权限，防止 Agent 自主执行高风险操作。

## 2. 用户评论数据集微调过程中的隐私保护

针对包含用户评论的数据集，隐私保护是重中之重，尤其要符合 GDPR 等法律法规。我会从以下几个具体步骤进行操作：

### 采取的措施与具体操作

#### 第一步：数据脱敏与匿名化

* **操作：** 在数据进入微调流程前，使用自动化脚本识别并剔除 PII（个人身份信息）。
* **具体内容：** 过滤掉评论中的真实姓名、手机号、身份证号、居住地址和社交账号。如果是必要的信息，采用“掩码”处理（如：张*三）。

#### 第二步：数据所有权与知情权

* **操作：** 遵循 GDPR 的“知情权”原则。
* **具体内容：** 明确告知用户这些数据将用于改进模型，并确保用户拥有“撤回”的权利。如果用户注销或要求删除数据，该数据必须从训练集中移除。

#### 第三步：审计与监控（留下“证据”）

* **操作：** 建立完善的审计日志（Audit Logs）。
* **具体内容：** 记录“谁（Who）、何时（When）、在何处（Where）、做了什么（What）”。例如，记录某位算法工程师在 2024-05-20 访问了该数据集进行微调。
* **目的：** 一旦发生泄露事件，可以快速溯源定责；同时在监管检查时，作为合规证明。

#### 第四步：最小权限原则

* **操作：** 严格限制能够接触到原始评论明文的人员数量。
* **具体内容：** 只有核心的“数据处理员”有权接触原始数据，模型训练人员仅能接触到脱敏后的数据集。
